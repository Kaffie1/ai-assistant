# 2026-04-26 编码习惯总结

这份文档只总结今天在重构过程中明确体现出来的编码偏好，用于后续继续收敛代码风格与架构边界。

## 1. 总体偏好

- 优先做最小可运行实现，不保留评估、治理、审计之类当前主链无关代码。
- 先删掉无关层和兼容层，再围绕当前真实需求重建结构。
- 代码要尽量直白，少兜圈，少为了“通用性”提前抽象。
- 只保留真正会被用到的接口，旧接口如果已经脱链就直接删除。

## 2. 结构体风格

- 全项目的结构体定义统一成 `Pydantic BaseModel + Field(...)` 风格。
- 尽量不要混用 `dataclass`、裸字典、半结构化对象。
- 状态对象、上下文对象、工具输入输出对象、事实对象都希望用同一套风格。
- 字段要有默认值和语义清楚的 `description`。

## 3. memory 目录设计

- memory 只保留明确分层，不要历史目录残留。
- 长期目标是围绕这些目录组织：
  - `models`
  - `store`
  - `long_term`
  - `short_term`
  - `persona`
- `persona` 就是个人画像，不应该混入 task/reminder store。
- `short_term` 只做短期会话读写，不承担 runtime 缓存策略。
- `store` 更适合承载 task/reminder 这类事实存储。

## 4. 短期记忆偏好

- 短期对话按天落文件。
- 文件名尽量简洁，只用日期。
- 存储格式用 `jsonl`，便于追加写，不做整文件重写。
- `recent_conversation.py` 对外只保留最少接口：
  - 一个 build/load
  - 一个 append/write
- `memory` 层只负责文件读写。
- “启动时加载一次、跨天再加载一次、平时走缓存”这类策略放在 runtime/workflow 层，而不是 memory 层。

## 5. 画像记忆偏好

- 画像单独存一个统一 JSON 文件。
- 画像内容只需要一行一行的文本，不需要复杂字段。
- `persona` 对外只保留最少接口：
  - build
  - append
- 工具里的画像操作要和 persona 文件走同一条链，不能再挂一套平行 store。

## 6. task / reminder 存储偏好

- task 和 reminder 要分文件维护，不要混在一个大文件里。
- 运行时可以先用内存对象，但需要有可读的持久化落地。
- 任务和提醒都希望落到 Markdown 文件中，方便排查和长期保留。
- `.md` 文件既是排查材料，也是最简持久化来源。

## 7. 工具系统偏好

- 工具模块自己声明自己的能力，不要在中心文件里维护越来越长的硬编码映射。
- 工具行为应该尽量下沉到 `ToolManifest` 或工具模块自身，例如：
  - `object_label`
  - `direct_response`
- 参数归一化函数也应该由工具模块自注册，而不是在 `tool_calling.py` 里写死一张 `tool -> normalizer` 表。
- 所有工具 handler 统一返回 `ToolResult`，不要同时兼容 `str` 和结构化结果。

## 8. LangGraph 偏好

- `graph.py` 只负责编排，不写具体 node 实现。
- node 实现都应该放在 `workflow/nodes/`。
- 如果流程里需要确认、中断、继续执行，就应该拆成明确节点，而不是在 runtime 里偷偷处理。
- 确认流更符合下面这种结构：
  - `tool_call`
  - `tool_confirm_prompt`
  - `tool_confirm`
  - `tool_execute`
- 中断提示文本应该由节点写入状态，不应在 `run_assistant_graph()` 里手工补状态。

## 9. runtime 偏好

- 能从主链剥离的 runtime 就尽量剥离。
- 如果某部分只是历史兼容层、已经不在图主链里，就可以删除。
- runtime 更适合保留“工具执行相关代码”这类尚未迁走的运行时能力，不适合继续承载主流程入口。

## 10. Web / CLI 偏好

- Web 和 CLI 都应尽量走同一套 workflow 主链。
- Web 自己不要再偷偷 new 一套和主流程不同的 store。
- 如果需要轮询提醒或通知，Web 也应该复用 workflow 正在使用的同一份事实存储。
- 启动方式希望统一，最好有一个简单启动脚本。

## 11. 注释风格

- 函数注释统一参考当前 workflow 风格：
  - `功能`
  - `输入`
  - `输出`
- 注释以帮助快速理解职责边界为主，不写低信息量废话。
- 复杂逻辑比起长篇解释，更偏好先把函数边界拆清楚。

## 12. 当前可直接复用的实现原则

- 先做最小链路通，再补文档和脚本。
- 改完以后要做 smoke test，至少覆盖：
  - 普通回复
  - 高风险确认提示
  - 取消恢复
  - 确认后执行
- 如果某块逻辑已经能通过节点拆分表达，就不要继续把补丁逻辑写在入口函数里。
